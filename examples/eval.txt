The measurement of computer performance is a discipline built on suspicion. A single number, reported once, tells you almost nothing: the machine that produced it was busy with other work, or the cache was warm when it should have been cold, or the clock stepped down because the room was hot. Anyone who has chased a benchmark regression across two supposedly identical servers learns the same lesson, which is that variance is not noise to be ignored but signal about everything you failed to control.

The honest approach is boring and repetitive. You fix the inputs: the same prompt, the same batch size, the same number of tokens, the same physical cores. You discard the first runs, because startup effects are real and nobody ships a cold binary a thousand times a second. You repeat the measurement enough times that the median means something, and you report an interval rather than a point, because the interval is the truth and the point is a summary of it. When two configurations differ, you interleave their runs so that whatever drifts in the background drifts over both of them equally.

Language models make this harder in an interesting way. Their inference cost is split between two phases with different bottlenecks. The prompt phase is a large matrix multiplication problem, hungry for arithmetic throughput, and it rewards wide vector units and clever kernels. The generation phase produces one token at a time, and each token requires reading the entire set of weights from memory, so the limit is bandwidth rather than arithmetic. A change that doubles one phase may do nothing for the other. Reporting a single tokens-per-second figure for the whole run hides exactly the distinction an engineer needs.

Quantization complicates the picture further. Shrinking weights from sixteen bits to four saves memory and bandwidth, and on the generation side this often translates directly into speed. But the compression is lossy, and the loss is not uniform: some layers tolerate it, some do not, and the degradation shows up not in the speed charts but in the quality of what the model writes. A tuning process that measures only speed will happily quantize a model into fluent nonsense and call it a win. The guard against this is to measure quality alongside speed on a fixed piece of text, and to refuse any configuration that trades one for the other past a stated threshold.

None of this is glamorous. It is bookkeeping, mostly, and discipline about what you claim. But the difference between a benchmark and an anecdote is precisely this bookkeeping, and the difference compounds: every decision made on top of a sloppy measurement inherits its sloppiness. Measure twice, report the interval, name the machine, pin the versions, and let anyone who doubts you rerun the whole thing with one command.

Consider what actually happens when a processor executes the generation phase of a language model. The weights of the network, several gigabytes of them even after compression, live in main memory. For every single token the model produces, essentially all of those weights must travel across the memory bus into the cores. The arithmetic itself, multiplying numbers and adding them up, is almost free by comparison; modern vector units can chew through the multiplications far faster than the memory system can deliver the operands. This is why the generation speed of a large model, measured in tokens per second, tracks memory bandwidth so closely that you can nearly predict one from the other with a pocket calculator: divide the achievable bandwidth by the size of the model in memory and you have an upper bound that no amount of kernel cleverness will beat.

The prompt phase inverts the economics. When the model ingests a long prompt, it can process many tokens in a single pass, which turns the workload from a sequence of thin matrix-vector products into a smaller number of fat matrix-matrix products. Now each weight fetched from memory is reused across many tokens, the arithmetic units become the bottleneck, and suddenly all the things that looked irrelevant during generation matter enormously: the width of the vector registers, the presence of specialized dot-product and integer matrix-multiply instructions, whether the kernel tiles its loops to keep operands resident in the nearest cache. Two machines with identical memory bandwidth can differ by a factor of four in prompt processing if one has matrix extensions the other lacks, or if one build of the software actually uses them and the other quietly does not.

That last clause deserves emphasis, because it is where a surprising amount of real-world performance is lost. Software distributed as prebuilt binaries must run on the oldest hardware it claims to support, so it is compiled for a lowest common denominator. The vendor's shiny instructions sit unused because the binary never asks for them. The fix costs nothing but knowledge: compile the same source on the machine that will run it, with the compiler told to target that machine, and features the portable build ignored light up. The speedup from this single change routinely exceeds the speedup from any amount of parameter tuning, and yet it is invisible to anyone who never questions the binary they downloaded.

Thread count is the parameter everyone reaches for first, and it is usually the least interesting. Doubling threads doubles neither bandwidth nor cache; it merely divides the same work more finely, and past the point where the memory system saturates, extra threads add coordination overhead and steal cache from one another. On machines with a single thread per physical core the folklore about leaving cores free for the operating system mostly transplants badly from other architectures. The honest answer, as always, is to measure: sweep the thread count, watch the curve flatten and then bend downward, and pin the setting at the knee rather than at the maximum.

The cache that holds attention state during generation, usually called the KV cache, offers a different trade. Stored at full precision it can consume more memory than a heavily compressed model itself once the context grows long. Halving its precision halves that footprint and the bandwidth spent streaming it, which directly accelerates long-context generation, but the cache is part of the model's working memory, and rounding it too aggressively degrades what the model remembers about its own context. The degradation is gradual and workload-dependent, which makes it exactly the kind of change that must be paired with a quality measurement rather than waved through on speed alone.

History offers a warning about every one of these knobs: the benchmark that flattered a change has a way of becoming the target that the change was optimized for. A configuration tuned until the benchmark improved has learned the benchmark, the way a student learns the past papers rather than the subject. The defense is pre-registration, the same discipline experimental sciences adopted for the same reason. Decide before the experiment what the baseline is, what the expert configuration to beat is, what threshold constitutes a win, and what measurement settles the question; write those decisions down where they cannot be quietly revised, and only then let the search run. Whatever the outcome, report it against the pre-registered target, including the uncomfortable cases where the expert loses to a machine that simply tried things and kept receipts.

There is a last habit that separates durable results from conference-demo numbers: reproduction on a clean machine. Every long tuning session accumulates state, in warmed caches, in leftover build artifacts, in environment variables somebody exported at hour three and forgot. A result that only manifests inside that accumulated state is not a result; it is an anecdote with a confidence interval. The remedy is to distill the winning configuration into a recipe, a short machine-readable description of exactly what to build and how to run it, then provision a fresh instance, replay the recipe with no search and no intelligence in the loop, and check that the number lands within tolerance of the claim. When it does, the claim has earned the only kind of trust that matters in this field: the kind that survives someone else's hardware, someone else's afternoon, and someone else's skepticism.

All of which is to say that performance work, done honestly, is less like wizardry and more like accounting with a stopwatch. The wins are real and sometimes large, but they are purchased with controls, repetition, intervals, pre-registration, quality guards, and reproduction, not with cleverness alone. The machine does not care how confident you feel. It answers exactly the question you actually asked it, which is why the craft consists almost entirely of learning to ask precisely.
