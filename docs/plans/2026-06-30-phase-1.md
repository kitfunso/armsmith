# Phase 1: Foundation + Spike 0 (GO/NO-GO) Implementation Plan

**Goal:** Stand up the repo foundation and pass Spike 0 (Performix MCP scriptability) - the GO/NO-GO gate - with an honest baseline benchmark captured on a real Graviton target.
**Prerequisites:** AWS account with Graviton access (interactive login); Python 3.11+; an Anthropic API key in env.
**Estimated scope:** 8 steps, ~1-2 working days (gated by AWS auth + Performix install).

> Phase 1 is deliberately Spike-0-first: the whole project's viability hinges on Performix being scriptable. We build only the minimum scaffold needed to reach the gate, then decide GO (MCP) vs NO-GO (CLI fallback) before investing in the loop (Phase 2).
>
> **Two parallel tracks (do NOT let one block the other):** the **backbone** (target up -> honest baseline -> build/bench/replay working) and the **Performix** track (Spike 0a/0b). A Performix install snag must not stall the backbone - prove llama.cpp build/bench/replay on Graviton regardless, then plug Performix in. The deterministic tuner is the core; the LLM analyst + Performix is the layer on top.

**Commit:** `chore: repo init, Apache-2.0 license, package skeleton`

## Step 3: Stand up a Graviton target (manual first - do NOT build the turnkey provisioner yet)
**Files:** `docs/spike0-result.md` (record the instance details + setup commands as you go)
**What:** Bring up a Graviton4 `r8g` (Ubuntu, spot) by hand via console/CLI, set an **auto-shutdown timer** on it (e.g. `sudo shutdown -h +180`) as a cost backstop, SSH in. The polished Terraform provisioner + `pipx` install path is deferred to post-GO (it is load-bearing for the <30-min reuse metric, but gold-plating it before the gate would waste the 46 days if Spike 0 is NO-GO).
**Verify:** you can `ssh` in and `uname -m` -> `aarch64`; the auto-shutdown timer is set.
**Commit:** `docs: spike0 graviton target setup notes`
**Note:** Requires interactive AWS auth. First paid step - tear down after every session.

## Step 4: Spike 0a - Performix profiling works
**Files:** `src/armsmith/profiler.py` (install + run helpers), `docs/spike0-result.md` (started)
**What:** Install Arm Performix on the target; profile a trivial workload (e.g. a llama.cpp run); capture counters to disk.
**Verify:** a `PerformixSnapshot`-shaped capture (cache miss, mem bw, IPC, hotspots) is written locally from a real run.
**Commit:** `spike: performix installs and profiles on graviton`

## Step 5: Spike 0b - Performix MCP scriptability (THE GATE)
**Files:** `src/armsmith/profiler.py` (MCP client path), `docs/spike0-result.md` (verdict)
**What:** Confirm the Performix MCP server can be driven headlessly from the control plane (not only from an IDE like Copilot/Kiro/Codex).
**Verify:** (a) a scripted (non-IDE) call to the MCP server returns structured perf data; AND (b) the counters **discriminate between levers** - profile two different builds (e.g. generic vs `-mcpu=native`) and confirm Performix clearly shows the difference (memory-bound vs compute-bound, cache/bandwidth deltas). Record both in `docs/spike0-result.md`.
**Decision:** **GO** = MCP scriptable AND counters discriminate -> Phase 2 uses Performix-driven diagnosis. **NO-GO on (a)** = parse Performix CLI output instead. **NO-GO on (b)** = keep Performix for headline metrics but reframe the loop as "try-measure-keep-best" rather than "diagnosis-driven" (still valid, less Performix-centric). Either way the project proceeds; this picks the profiler path and the loop framing.
**Commit:** `spike: performix MCP scriptability verdict (GO/NO-GO)`

## Step 6: Honest baseline benchmark
**Files:** `src/armsmith/bench.py`, `src/armsmith/engine.py` (llama.cpp build/run), `src/armsmith/models.py` (`BenchmarkResult`)
**What:** Build a stock (honest naive) llama.cpp on the target; **also pin the pre-registered expert config** (hand-tuned, recorded now so the >=90%-gap target can't be reverse-engineered later). Run an 8B-class model (e.g. Llama-3.x 8B Q4_0) by **wrapping `llama-bench`** (do NOT reimplement). Capture: decode tok/s + prefill TTFT, **and a quality number (perplexity/KL)** so later quant changes can't win by degrading the model. **Two-stage**: a cheap screen pass, then the rigorous N>=5 + 95% CI confirm only for candidates that beat the screen. Measurement integrity: **A/B/A/B interleaving**, thread pinning, non-burstable instance, fixed prompt/batch/context, instance/kernel/governor in the manifest.
**Verify:** `trajectories/<run_id>/baseline.json` (and `expert.json`) have decode_tok_s + prefill_ttft with CIs, a quality value, n_repeats>=5, and the captured environment; numbers stable across two invocations (<10% drift).
**Commit:** `feat: rigorous benchmark harness + first honest Graviton baseline`

## Step 7: Action space registry (stubs)
**Files:** `src/armsmith/actions.py`, `tests/test_actions.py`
**What:** Define the `ActionSpec` registry with the **5 levers** (`-mcpu=native`, KleidiAI, quant format, threading/affinity, KV-cache) as apply/revert command templates + preconditions + a param schema per action. (A plain "generic build" is the baseline/reset state, NOT an action.) Add the **validator** that accepts an LLM `{action_id, params}` suggestion only if the id is in the registry and params match the schema. No loop yet.
**Verify:** `pytest tests/test_actions.py` passes: every action has apply/revert/precondition; registry lists exactly 5 actions; **and the CRITICAL safety test - an off-registry `action_id` (or out-of-schema param) is rejected and never reaches the executor**.
**Commit:** `feat: action space registry + validator (stubs + safety test)`

## Step 8: Generate Phase 2 plan
**Files:** `docs/plans/<date>-phase-2.md`
**What:** Write the Phase 2 plan: the **discovery** loop (deterministic tuner pops candidates; LLM analyst reorders the queue from Performix evidence + writes rationale; two-stage bench; keep/revert on CI-significant speed AND no quality regression; trajectory), saving a deterministic **recipe**; then the **visual-trajectory HTML report** (baseline -> diagnosis -> action -> delta -> keep/revert -> recipe, showing the machine reasoning, not log dumps); then `repro` (replays the recipe, no LLM, pinned binaries, <30 min); then the turnkey provisioner + `pipx` install path (deferred from Step 3, gated on GO).
**Verify:** `docs/plans/<date>-phase-2.md` exists with verifiable steps.
**Commit:** `docs: phase 2 plan (agent loop)`
